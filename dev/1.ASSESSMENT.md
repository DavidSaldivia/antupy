# antupy Independent Assessment (June 2026)

## Scope and Method

This assessment was prepared from a direct pass through:

- project root metadata and governance files
- documentation sources and generated docs
- package source under `antupy/`
- paper artifacts under `papers/`
- tests under `tests/` and `testing/`

This is an independent view. Any existing roadmap/planning files were not used as the authority for recommendations.

## Executive Assessment

`antupy` is not a scripts folder project. It is a real, architected research library with coherent abstractions and non-trivial test coverage.

Current maturity score (0-10 scale): **6.4/10**.

Interpretation:

- 0-2: ad-hoc scripts
- 3-4: alpha code with limited structure
- 5-6: early library with solid foundations
- 7-8: robust open-source library
- 9-10: ecosystem-level maturity (NumPy/Polars class)

`antupy` sits in the **"solid early library"** band.

## Why 6.4/10 (Dimension Scores)

- Architecture and API design: **8.0/10**
- Core engineering quality (typing, design, modularity): **7.5/10**
- Testing posture: **6.5/10**
- Documentation quality: **6.5/10**
- Release and CI discipline: **4.5/10**
- Collaboration/governance readiness: **4.0/10**

Weighted combined score: **6.4/10**.

## What Is Already Strong

- Clear layered design (core, utilities, simulation/TSG).
- Good foundational domain objects (`Var`, `Array`, `Frame`) with unit-aware operations.
- Public API export is clean and discoverable.
- Rich domain intent and motivation in paper material.
- Non-trivial test suite breadth (units, frame, props, weather, plant/parametric).
- Type hints and Protocol-driven design are present across major modules.

## Main Gaps Holding Back Quality Tier

### 1) Release/docs trust gap

- Package version in `pyproject.toml` is `0.6.2`.
- Changelog has duplicated `0.5.0` sections and no clear `0.6.x` release narrative.
- Generated docs in `docs/build/html` still display `0.5.0`.

Impact: users cannot reliably infer what version the docs represent.

### 2) Collaboration process gap

- No formal contributor protocol (PR contract, ownership model, review policy, release checklist).
- `DEVS.md` has useful intent but is not enough for onboarding multiple collaborators.

Impact: contributor throughput and consistency will degrade as team grows.

### 3) CI automation gap

- No visible test/lint/type CI workflow in `.github/workflows` (only a paper-related workflow).
- Quality expectations exist but are not machine-enforced.

Impact: regression risk and merge friction increase with every collaborator.

### 4) Documentation alignment gap

- README frames `Frame` as `polars(pandas)` while docs describe a pandas-backed implementation.
- Some docs language and examples still read as transitional.

Impact: contributor and user confusion about current backend reality and roadmap state.

### 5) Consistency/maintenance signals in code

- Mixed naming/legacy API traces in utility modules (for example method naming expectations such as `mu`/`viscosity` patterns).
- This is manageable now but will become a maintenance tax with multiple authors.

## Direct Answers to Your Two Specific Questions

## Q1. PyPI is 0.6.2 but docs are 0.5.0. How to update?

Root cause is primarily stale generated docs artifacts and release/version discipline, not only source docs.

Recommended approach:

1. Define canonical release source as `pyproject.toml` version.
2. Update changelog to include proper `0.6.2` (and any missing `0.6.x`) entries.
3. Rebuild docs from `docs/source` against the current package version.
4. Publish docs from source build pipeline (Read the Docs or GH Actions), not from stale committed HTML.
5. Decide policy for `docs/build/`:
   - preferred: do not version-control generated HTML
   - if kept: regenerate on each release and enforce consistency check.
6. After release/tag, verify docs banner/version string matches package release.

Short answer: treat docs as a build artifact tied to release/tag, not manual snapshots.

## Q2. README and docs inconsistencies

Most important mismatch:

- README implies `Frame` is already in a `polars(pandas)` state.
- Docs and implementation are pandas-centric with unit tracking extensions.

Additional alignment targets:

- Standardize naming for simulation abstractions (`Analyser`/`Parametric` wording).
- Keep one canonical terminology table and reuse in README + docs intro.
- Ensure examples use the same current API style as tests.

## Collaboration Readiness Snapshot

Current readiness for multiple external collaborators: **moderate-low**.

What is needed first:

- a concise contributor protocol
- CI gate on tests + typing + lint
- clear branch/review/release conventions
- docs/release synchronization policy

Once those are in place, the codebase quality is strong enough to scale contributors.

## Bottom Line

`antupy` is a credible pre-1.0 scientific library with strong technical DNA and clear domain value. The fastest path from 6.4 toward 7.5+ is not rewriting core physics logic; it is tightening release discipline, collaboration protocol, and automation around what already works.
